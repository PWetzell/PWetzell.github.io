Research · Information architecture

Defining the structure before designing the interface.

Two IA deliverables produced from customer interview findings — node hierarchy architecture and system relationship diagram — completed before any screen design began. Both directly informed the Configure and Investigate tab structure in the final product.

Artifact 04 · Site node hierarchy
How the Configure and Investigate tabs needed to be structured.
The IA mapped the full node hierarchy — Account → Region → Site → Building → Floor → Area → Asset → Device — across both the Configure and Investigate tab contexts. Each level carried different data aggregation rules and action permissions that had to be reflected in the UI before any component was designed.
Design direction set
Tab structure had to reflect the node hierarchy, not the navigation taxonomy. Users needed to operate at any level of the hierarchy without losing context of where they were in the tree.
Artifact 05 · System relationship diagram
How Tomo, IDMS, Exchange, and EEO connected across single and multi-account scenarios.
Two system diagrams mapped the difference between a single-account activation (one IDMS, one Exchange connection) and a multi-account scenario (multiple IDMS instances, multiple EEO products). This distinction was invisible in the existing UI but caused the majority of activation failures discovered in research.
Design direction set
The activation flow had to account for multi-account scenarios from day one. A single generic flow was not viable — territory and account context had to be resolved before any module selection step.
