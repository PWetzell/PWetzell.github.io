The Problem

Design-to-dev handoff has always been a translation step. And translations always lose something.

After 12+ years on enterprise products I've built design systems, watched them drift, and rebuilt them. It's never a discipline problem — it's an architecture problem. Design and code are always two separate artifacts that have to be manually kept in sync. I wanted to know if new tooling could finally make that unnecessary.

The question I kept coming back to

Why can't the design effort translate directly into something developers can consume — scalable, consistent, with no translator sitting in between?

Two technologies arrived that I believed could finally close this gap: the Figma Plugin API, which lets you generate and manipulate design files programmatically, and AI tooling capable of writing production-quality code from design intent. I didn't want to read about them. I needed a real project to find out.

The Approach

One file as the source of truth. Everything else generated from it.

I wrote a Figma plugin that generates the entire Acme Design System component library programmatically from a single HTML master file — a living style guide that defines every token, variant, and state. Run the plugin and it rebuilds the Figma library from scratch. The same tokens feed a developer-ready CSS file. There's nothing to keep in sync because there's only one thing to change.

Feedback loop
Figma MCP
Live read · drift detection

Figma MCP closes the feedback loop: when you tweak a value directly in Figma, MCP reads it back out live — exact fills, font sizes, padding — so it can be codified into the source file. One edit, one command, every component updates. Nothing drifts because nothing is maintained separately.

Token architecture

Contracts, not descriptions

blue-500 breaks when a second product arrives. --color-brand-primary doesn't. Three-tier hierarchy: global values → semantic aliases → component tokens. Framework-agnostic CSS — React, Angular, .NET Blazor, Vue, plain HTML.

AG Grid — the hardest component

5 density variants + full controls

Most systems stop at the grid. I built the toolbar, column chooser, saved views, and all header states too — because that's where users actually live. Token names map directly to AG Grid's CSS variable system. No lookup table.

Design-to-code fluency

Catching constraints before they become engineering problems

Writing 5,700+ lines of plugin code — JavaScript against the Figma Plugin API — changed how I design. I know how component properties map to code, where CSS variable naming breaks down at scale, and which design decisions create implementation debt. On collaborative projects, that means catching constraint mismatches in design review rather than QA — saving engineering cycles and reducing back-and-forth on handoff gaps.

Framework-agnostic output

One source, any stack

The plugin generates CSS custom properties that work identically in React, Angular, .NET Blazor, Vue, or plain HTML — because the token architecture is decoupled from any specific framework's rendering model. This is the same principle that makes design systems scale across engineering teams without fragmentation.

AI + Experience

AI handled implementation velocity. Experience determined the quality of what got built.

Where AI fell short
Its first pass at component variants stacked them positionally — valid code, wrong architecture. I recognized it immediately because I'd seen what happens when variant structure doesn't reflect semantic intent.
Where experience corrected it
Token naming defaulted to descriptive — blue-500 style. I pushed it to semantic contracts because I'd watched descriptive naming break down across a product family.
The actual skill shift
The constraint moved from "can I write this?" to "do I know what to ask for?" Twelve years of shipping enterprise products is what answered that second question correctly.

"The experience doesn't become less relevant when AI writes the code. It becomes the differentiator."

What I'd do differently

Start with the AG Grid theme, not the button components. Mapping tokens to AG Grid's 200+ CSS variables forces every important architectural decision first — what the spacing scale needs to cover, what "density" means as a system property, which semantic tokens are necessary versus convenient. The hardest constraint should be the first constraint.
