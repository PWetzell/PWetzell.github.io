The Problem

FINRA investigators buried under fragmented tools, siloed data, and manual reconciliation — with no unified view of an active investigation.

FINRA investigators tracking financial crimes were working across disconnected systems — case management, data browsing, surveillance tools, and manual Excel workflows — with no single workspace to bring it together. Key signals were buried, context switching was constant, and the cognitive load of managing an investigation was slowing down the work of protecting markets.

As part of FINRA's Digital Experience Transformation (DXT), I helped design the Financial Crimes Investigation Dashboard — a unified, widget-based investigative workspace built on FINRA's Gateway platform, backed by AWS cloud infrastructure and Angular frontends with .NET microservices.

The core tension

Investigators needed the depth of six specialized tools in a single workspace — without losing the precision any of those tools provided individually.

Consolidation isn't just a UI problem — it's an information architecture problem. Every investigator had a different workflow, different data priorities, and different definitions of what "fast access" meant. The widget-based dashboard model had to be modular enough to serve all of them, while structured enough to guide decision-making under pressure.

My Role

What I owned
Full UX scope — contextual inquiry, workflow mapping, investigative flow architecture, widget component design, interaction fidelity prototyping, fast validation testing, and visual clarity refinement.
The platform context
Designed for FINRA's Gateway — a widget-centric dashboard framework on AWS. Components were built as modular, interchangeable widgets consumable across the platform's Angular frontend and .NET microservices backend.
The research depth
This project was grounded in real investigator workflows — contextual inquiry across multiple teams, workflow architecture mapping, and iterative prototype testing. The design was shaped by evidence, not assumptions.
Process

Five phases — from contextual inquiry and workflow mapping to interaction fidelity and fast validation.

Partnered with FINRA investigators, product, and engineering to understand the full investigative workflow, map decision points and friction, and design a modular workspace that surfaced the right signal at the right moment.

01
Contextual Inquiry
Conducted structured contextual inquiry across investigator teams — surfacing the most-requested features by team, mapping where current tools created the most friction, and identifying the signals investigators needed fastest. The data directly shaped prioritization decisions.
02
Workflow Architecture & Investigative Mapping
Built a detailed Matter Sequence workflow map — tracing the full investigative lifecycle from case intake through resolution. Identified branching decision points, data dependencies, and the moments where investigators needed consolidated context most urgently.
03
Structured Collaboration & Widget Architecture
Ran cross-functional sessions to align on the widget-based dashboard model — defining which investigative functions became modular widgets, how they connected, and how the Gateway platform's component architecture would support customizable investigator workspaces.
04
Interaction Fidelity & Prototyping
Built high-fidelity prototypes covering the full investigator journey — case overview, data browsing, cross-reference navigation, alert triage, and collaborative annotation. Prototypes were detailed enough to test real investigator behavior, not just surface impressions.
05
Fast Validation & Visual Clarity
Ran rapid usability testing with investigators to validate triage logic, information hierarchy, and widget layout decisions. Iterated on visual clarity — density, color coding, progressive disclosure — until the dashboard could be read accurately under time pressure.

Widget-based architecture

Modular by design — customizable by investigator

The dashboard was built on FINRA Gateway's widget framework — each investigative function (case summary, alert feed, data cross-reference, timeline) existed as an independent, interchangeable component. Investigators could configure their workspace around their specific workflow without losing system coherence.

Research-driven decisions

Every priority traceable to investigator evidence

The contextual inquiry data directly drove feature prioritization, widget hierarchy, and information density decisions. Nothing in the final design was arbitrary — every choice mapped back to a documented investigator need or workflow constraint.

"The most important design decision wasn't visual — it was defining which investigative functions needed to live together in the same view, and which ones could stay a click away."

Outcomes

A unified investigative workspace — replacing fragmented tools with a governed, widget-based platform built for financial crimes work.

Consolidated 6+ investigative tools into a single, configurable workspace — eliminating the context switching that slowed investigator decision-making.
Widget-based architecture adopted across FINRA Gateway — modular components that investigators could configure around their specific workflow needs.
Research-validated design decisions — every feature priority, widget hierarchy, and density choice traceable to documented investigator evidence from contextual inquiry.
Faster alert triage and case review — structured risk indicators and progressive disclosure replaced manual cross-referencing across disconnected tools.
AWS cloud-native scalability — the dashboard architecture was built to scale with FINRA's real-time data processing requirements, not against them.
Design system contribution — investigative widget patterns contributed to FINRA Gateway's centralized component library, ensuring consistency across the consolidated platform.

What I'd do differently

Start widget governance conversations with engineering earlier. The modular architecture worked well, but some widget interaction patterns — particularly cross-widget data sharing and state persistence — weren't fully defined until mid-build. A joint design-engineering session on widget communication contracts at the start of the project would have prevented the rework that came later. When the component architecture is this central to the product, the technical model and the design model need to be designed together from day one.
